# UUIDv7 Audit — Phase 1 Status Report

## Scope

- **Scanned:** `lib/services/*`, `app/api/*`, `pages/api/*`, and application codebase for UUID generation.

## Findings

### Directories

| Path              | Exists | Notes                    |
|-------------------|--------|--------------------------|
| `lib/services/*`  | **No** | Directory not present    |
| `app/api/*`       | **No** | No API routes directory  |
| `pages/api/*`     | **No** | No pages API directory   |

### Application code: no explicit UUIDv4 calls

**No usages found** of:

- `crypto.randomUUID()`
- `uuid()` (generic)
- `v4()` from the `uuid` package
- Any other in-code UUID generation

### Where IDs are actually assigned

Primary keys are currently generated by **PostgreSQL** via `DEFAULT gen_random_uuid()` (UUIDv4) in:

`supabase/migrations/20260219100000_create_base_tables.sql`:

- `public.subxeuron` — `id uuid PRIMARY KEY DEFAULT gen_random_uuid()`
- `public.post` — `id uuid PRIMARY KEY DEFAULT gen_random_uuid()`
- `public.comment` — `id uuid PRIMARY KEY DEFAULT gen_random_uuid()`
- `public.vote` — `id uuid PRIMARY KEY DEFAULT gen_random_uuid()`

`public.user` uses `id uuid PRIMARY KEY REFERENCES auth.users(id)` — ID comes from Supabase Auth, not from app or DB default.

### Insert call sites (relying on DB default = UUIDv4)

| File | Function / location | Table | Passes `id`? |
|------|--------------------|-------|--------------|
| `lib/supabase/mutations.ts` | `createSubxeuron` | subxeuron | No → DB default |
| `lib/supabase/mutations.ts` | `createPost` | post | No → DB default |
| `lib/supabase/mutations.ts` | `addComment` | comment | No → DB default |
| `lib/supabase/votes.ts` | `upvotePost` | vote | No → DB default |
| `lib/supabase/votes.ts` | `downvotePost` | vote | No → DB default |
| `lib/supabase/votes.ts` | `upvoteComment` | vote | No → DB default |
| `lib/supabase/votes.ts` | `downvoteComment` | vote | No → DB default |
| `lib/supabase/user.ts` | `getUser` (insert on first login) | user | Yes — `id: authUser.id` (Auth UUID) |

## Conclusion

- **No** application code calls `crypto.randomUUID()`, `uuid()`, or `v4()`.
- The regression is **reliance on DB default** `gen_random_uuid()` (UUIDv4) for **subxeuron**, **post**, **comment**, and **vote** inserts.
- To enforce UUIDv7, the application must pass an explicit `id` generated with UUIDv7 at each of these insert sites; the DB default remains as a fallback for ad‑hoc SQL.

# Phase 3 — UUIDv7 as Default (Post pnpm install)

## Completed

1. **DB default set to UUIDv7**  
   Migration `supabase/migrations/20260222150000_add_uuidv7_default.sql`:
   - Ensures `pgcrypto` (for `gen_random_bytes`).
   - Defines `public.gen_uuidv7()` (RFC 9562–style UUIDv7 in PL/pgSQL).
   - Sets `DEFAULT public.gen_uuidv7()` on `public.subxeuron`, `public.post`, `public.comment`, `public.vote`.  
   - **Not** applied to `public.user` (id comes from `auth.users`).

2. **ESLint**  
   In `eslint.config.mjs`:
   - `no-restricted-imports`: forbid importing `v7` from `'uuid'` except in `lib/utils/id.ts`.
   - Only `lib/utils/id.ts` may call `uuidv7()`; all other code must use `generateId()` from `@/lib/utils/id`.

## Apply the migration

From the project root:

```bash
npx supabase db push
```

Or, if using Supabase CLI migrations:

```bash
npx supabase migration up
```

After this, any insert that does **not** supply `id` (e.g. ad‑hoc SQL or future code paths) will get a UUIDv7 from `gen_uuidv7()`. Application inserts continue to pass `id: generateId()` from `lib/utils/id.ts`.
